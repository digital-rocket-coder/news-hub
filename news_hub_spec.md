# ТЗ: Персональный новостной хаб («второй мозг» для новостей)

> Документ-задание для реализации приложения. Описывает продукт, философию, функции (с разделением на MVP и расширения), модель данных, ключевую логику и открытые развилки.

---

## 1. Суть продукта

Персональный новостной агрегатор, который превращает поток RSS-новостей в **структурированную базу знаний по темам**. Приложение само собирает важные новости из заданных источников и хранит их, пока пользователь не будет готов читать. Главное отличие от обычных RSS-читалок — упор не на бесконечную ленту, а на осмысленное изучение тем: со сводками, динамикой популярности тем и визуальным графом для исследования материала.

**Питч одной фразой:** RSS-читалка, которая работает не как лента, а как карта тем — копит новости, показывает тренды и даёт изучать материал через граф, как в Obsidian.

---

## 2. Философия (важно соблюдать при реализации)

Эти принципы определяют все решения. Если фича им противоречит — её не добавляем.

- **Pull, а не push.** Приложение не дёргает уведомлениями. Оно спокойно копит контент, пользователь приходит сам, когда готов.
- **Темы, а не хронология.** Базовая единица навигации — тема, а не дата. Чтение идёт по смыслу.
- **Изучение, а не потребление.** Цель — не «успеть прочитать всё», а разобраться в том, что важно. Продукт снимает тревогу непрочитанного.
- **Персональность.** Это личный инструмент. Никаких соцфункций, лент друзей, шаринга — по крайней мере на старте.

---

## 3. Основной цикл использования

1. **Фон.** Приложение постоянно тянет новости из источников и раскладывает их по темам — без уведомлений.
2. **Возвращение.** Пользователь заходит, когда готов. Первым видит не сырую ленту, а сводку: «вот что важного накопилось».
3. **Погружение.** Если тема цепляет — пользователь проваливается в неё через граф и читает всё, что под ней собралось, в логике смысла.
4. **Картина во времени.** Параллельно видит, какие темы растут, а какие затухают.

## 4. Функции — MVP (первая очередь)

Минимум, без которого продукт не имеет смысла.

### 4.1. Агрегация RSS
- Пользователь добавляет RSS-источники (URL фида).
- Приложение в фоне периодически опрашивает источники и подтягивает новые материалы.
- Ничего не теряется, даже если пользователь долго не заходил.
- У каждого источника можно задать тег/категорию вручную.

### 4.2. Темы (тематики)
- Каждая новость относится к одной или нескольким темам.
- **Ядро — эмбеддинги** (решено): для каждой новости считается векторный эмбеддинг; темы формируются кластеризацией близких новостей. Этим же механизмом строятся связи графа и поиск похожего.
- **Названия тем** генерирует LLM по содержимому кластера.
- **Режим на старте — полуавтоматический:** приложение предлагает кластеры тем, пользователь их подтверждает и переименовывает.
- **Ручной override:** пользователь может задать тему как тег + ключевые слова и вручную привязывать новости.
- См. раздел 7.1 о деталях.

### 4.3. Лента по темам
- Просмотр новостей, сгруппированных по темам (не по дате).
- Внутри темы — список материалов с заголовком, источником, датой, кратким описанием.
- Отметка прочитано/непрочитано.

### 4.4. Дайджест
- Сводка самого главного за период (по умолчанию неделя).
- **Это отбор главного, а не список всех новостей.** Принцип: топ-N тем по динамике и объёму, под каждой 1–3 ключевых материала.
- Должен умещаться в пару минут чтения и отвечать на вопрос «что я пропустил важного».
- **Адаптивность по сроку отсутствия:** пропал на день — короткая сводка; пропал на месяц — расширенный обзор с разбивкой по неделям.

### 4.5. Динамика тем
- Для каждой темы считается «вес» по периодам (например, по неделям) — сколько важных новостей в неё попало.
- На основе веса — тренд: растёт / падает / стабильна / новая (всплеск).
- Минимальная визуализация: стрелка ↑↓ рядом с темой + детальный график при клике.

### 4.6. Граф тем
- Визуальная паутинка (force-directed граф, как в Obsidian).
- Узлы = темы. Размер/яркость узла = текущая значимость темы.
- Связи между узлами = смысловая близость (темы, часто встречающиеся вместе в новостях).
- **Клик по узлу раскрывает все материалы под темой.** Это основной режим «изучения»: от карты к статьям.

## 5. Функции — расширения (вторая очередь)

Усиливают идею, но не обязательны для первого работающего варианта.

### Углубление графа
- **Таймлайн сюжета:** связи не только между темами, но и между новостями — «развитие этого сюжета». Внутри темы видно, как событие развивалось от первой публикации до последствий.
- **Соседние темы:** при клике на тему граф подсвечивает смежные — то, что часто упоминается вместе.
- **История графа во времени:** ползунок, проигрывающий, как карта тем менялась за месяц (что выросло, что появилось).

### Работа с контентом
- **Саммари:** краткая выжимка каждой статьи (1–2 абзаца) + сводное саммари по теме целиком.
- **Дедупликация:** схлопывание одного события из нескольких источников в «кластер-событие» с пометкой «освещено в N источниках»; подсветка расхождений в трактовке.
- **Уровни глубины:** для темы три слоя — «заголовки» / «суть» (саммари) / «полностью» (оригиналы).

### Приоритеты
- **Обучение на поведении:** приложение замечает, какие темы пользователь реально открывает и дочитывает, и подстраивает порядок в дайджесте. Без явных настроек.
- **Закладки и заметки:** сохранить новость на отдельную полку и оставить свою заметку (превращает в настоящий «второй мозг»).
- **Заглушение тем:** пометить темы, которые не хочется видеть, чтобы они не засоряли граф и дайджест.

### Сценарии возвращения
- **«С чего начать»:** после долгого перерыва приложение показывает «вот 3 темы, которые сильнее всего изменились, пока тебя не было».

### Источники (расширение)
- На MVP — только RSS.
- **Слой загрузки источников должен быть расширяемым** (абстракция «коннектор источника»), чтобы потом добавить новые типы без переписывания.
- В перспективе: **Twitter/X**, Telegram-каналы, сайты без RSS, рассылки.
- ⚠️ Нюанс: у Twitter/X API сейчас платный и ограниченный доступ — учесть при планировании этой интеграции.

## 6. Предлагаемая модель данных

Хранилище: **PostgreSQL + pgvector** (решено). Реляционные данные и векторы эмбеддингов — в одной БД.

- **Source** — источник: `id`, `тип` (rss / в перспективе twitter, telegram…), `url`, `название`, `тег/категория`, `интервал опроса`, `время последнего опроса`.
- **Article** — новость: `id`, `source_id`, `заголовок`, `ссылка`, `опубликовано`, `получено`, `полный_текст/описание`, `embedding` (vector, pgvector), `саммари` (опц.), `статус` (прочитано/нет), `закладка` (опц.).
- **Topic** — тема: `id`, `название` (генерит LLM), `тип` (авто/ручная), `embedding` (vector, центроид кластера), `ключевые слова` (для ручного override), `заглушена` (bool).
- **ArticleTopic** — связь many-to-many: `article_id`, `topic_id`, `вес/уверенность`.
- **TopicTrend** — динамика: `topic_id`, `период` (дата/неделя), `вес` (число новостей или метрика).
- **TopicLink** — связь тем в графе: `topic_a`, `topic_b`, `сила связи` (косинусная близость и/или co-occurrence).
- **UserNote** (расширение): `article_id`, `текст`, `создано`.

## 7. Ключевая логика (на что обратить внимание разработчику)

### 7.1. Как определять темы — решение принято
**Ядро — эмбеддинги:**
1. Для каждой новости считаем эмбеддинг (внешний API).
2. Кластеризуем близкие новости → каждый кластер = тема-кандидат.
3. LLM генерирует человекочитаемое название темы по содержимому кластера.
4. Пользователь подтверждает/переименовывает/объединяет кластеры.
5. Связь графа и поиск похожего — через косинусную близость эмбеддингов.

**Ручной override:** пользователь может создать тему вручную (тег + ключевые слова) и прибивать к ней новости.

Эмбеддинги выбраны ядром потому, что одним механизмом закрывают кластеризацию тем, связи графа и поиск похожего.

### 7.2. Что считается «важным»
Комбинация: (а) явный приоритет пользователя (отмеченные темы/источники) ∪ (б) объективная значимость — частота и резкость роста темы в источниках. Стоит вынести в конфигурируемую формулу.

### 7.3. Логика тренда темы
- Считать вес темы за окно (например, скользящие 7 дней).
- Сравнивать с предыдущим окном → растёт/падает.
- «Новая/всплеск» — если тема появилась недавно и сразу набрала вес.

### 7.4. Связи в графе
- Сила связи между двумя темами ∝ как часто новости относятся к обеим одновременно (co-occurrence) либо близости эмбеддингов тем.

## 8. Технологический стек (решено для веб-MVP)

- **Тип приложения:** веб-приложение.
- **Фронтенд:** React. Для графа — библиотека force-directed визуализации (например, react-force-graph / Cytoscape.js / D3).
- **Бэкенд:** Python + FastAPI (сильная экосистема для парсинга фидов и работы с моделями). Альтернатива — Node.js, если так удобнее.
- **База данных:** PostgreSQL + pgvector (реляционные данные + векторы эмбеддингов в одном месте; аналитика трендов — SQL-запросами).
- **Парсинг RSS:** библиотека парсинга фидов (например, feedparser для Python).
- **Фоновый опрос источников:** планировщик/воркер (например, APScheduler или Celery), периодически обновляющий фиды.
- **Эмбеддинги:** внешний API (вызов из бэкенда).
- **Саммари и названия тем:** внешняя LLM (Claude), вызов из бэкенда.

Локальные модели — возможная замена внешним API позже, если важна приватность или стоимость.

---

## 9. Анти-фичи (осознанно НЕ делаем)

- **Push-уведомления** в их обычном виде — противоречат принципу pull. Максимум — один опциональный мягкий сигнал «накопилось что-то крупное», без ежедневного дёрганья.
- **Соцфункции** (шаринг, лента друзей, комментарии) — превращают спокойный инструмент в тревожную соцсеть. Держим продукт персональным.
- **Бесконечная хронологическая лента** как основной режим — навигация строится вокруг тем, а не даты.

---

## 10. Зафиксированные решения

1. **Платформа:** веб-приложение (MVP).
2. **Определение тем:** ядро — эмбеддинги + LLM для названий; ручной override по ключевым словам.
3. **Хранилище:** PostgreSQL + pgvector (данные + аналитика + векторы в одной БД).
4. **Модели:** внешние API (эмбеддинги + Claude для саммари/названий), вызов из бэкенда. Локальные — позже при необходимости.
5. **Источники:** на MVP только RSS; слой источников расширяемый. В перспективе — Twitter/X, Telegram, сайты без RSS, рассылки (у Twitter/X — платный/ограниченный API).

### Осталось определить по ходу
- Аутентификация: один пользователь или мультиюзер? (для личного MVP можно начать без полноценной авторизации)
- Хостинг и развёртывание.
- Бюджет на вызовы внешних моделей (влияет на частоту пересчёта эмбеддингов/саммари).

---

### Краткое резюме для реализующего
Построить **веб-приложение** — персональный RSS-агрегатор, где навигация идёт по темам, а не по ленте. Стек: **React + FastAPI (Python) + PostgreSQL/pgvector**, внешние API для эмбеддингов и LLM. Минимум: сбор фидов в фоне → эмбеддинги → кластеризация в темы → дайджест главного → динамика тем → граф для изучения. Соблюдать философию pull/темы/изучение/персональность. Слой источников делать расширяемым (RSS сейчас, Twitter/X и др. потом). Не добавлять push и соцфункции.
